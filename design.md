# Design Document — internship-listing-page

## Overview

InternTrail is a static, single-page web application that surfaces fresher-friendly internship listings. The entire application lives in three files: `index.html`, `style.css`, and `script.js`. There is no backend, no build step, and no external dependencies.

The page presents a curated set of internship cards in a responsive grid. Users can filter by work type (Remote / Onsite). Clicking any card opens a detail modal with full role information and a direct apply link. The design is mobile-first and must be fully keyboard and screen-reader accessible.

### Key Design Decisions

- **No framework** — vanilla HTML/CSS/JS keeps the project approachable for student contributors and removes all toolchain complexity.
- **Hardcoded data** — `internships` array in `script.js` acts as the single source of truth; no fetch calls, no async concerns.
- **CSS custom properties** — all design tokens (colours, spacing, font sizes) live in `:root` so the visual theme can be adjusted in one place.
- **Modal over page navigation** — a `<dialog>`-based overlay keeps the user on the same page and avoids unnecessary history entries.
- **CSS Grid + mobile-first media queries** — the simplest, most maintainable approach for a 1 → 2 → 3 column responsive layout.

## Architecture

The application follows a simple data → render → event flow with no shared mutable state beyond two module-level variables.

```
┌─────────────────────────────────────────────────────┐
│                     index.html                      │
│  Structural skeleton: header, nav, main, footer     │
│  Links style.css and script.js                      │
└──────────────────┬──────────────────────────────────┘
                   │ DOM
          ┌────────▼────────┐
          │   script.js     │
          │                 │
          │  [DATA]         │  internships[] — hardcoded array
          │  [RENDER]       │  renderCards(filter) → writes to #card-grid
          │                 │  openModal(id)      → populates + shows <dialog>
          │                 │  closeModal()       → hides <dialog>, returns focus
          │  [EVENTS]       │  filter buttons, card "View Details", close button,
          │                 │  Escape key, backdrop click
          └────────┬────────┘
                   │ reads / writes CSS classes & aria attributes
          ┌────────▼────────┐
          │   style.css     │
          │  :root tokens   │
          │  mobile-first   │
          │  Grid layout    │
          │  Modal overlay  │
          └─────────────────┘
```

### State

| Variable | Type | Purpose |
|---|---|---|
| `currentFilter` | `string` | Active filter value: `"all"`, `"Remote"`, or `"Onsite"` |
| `lastFocusedCard` | `Element \| null` | Reference to the "View Details" button that opened the modal, so focus can be restored on close |

No other mutable state exists. The rendered DOM is always a pure function of `internships` filtered by `currentFilter`.

## Components and Interfaces

### 1. Page Shell (`index.html`)

Provides the static structure that JavaScript populates.

```
<header>          — brand name + tagline
<nav>             — Filter_Bar with three <button> elements
<main>
  <div id="card-grid">   — Card_Grid (populated by JS)
  <dialog id="modal">    — Detail_Modal (populated by JS on open)
<footer>          — page-level footer
```

Key attributes set in HTML (not JS):
- `<dialog>` starts without the `open` attribute; JS calls `.showModal()` / `.close()`.
- Each filter `<button>` has `data-filter="all|Remote|Onsite"` and `aria-pressed="false"` as defaults.

### 2. Card (generated by `renderCards`)

Each card is an `<article>` element injected into `#card-grid`.

```
<article class="card" data-id="{id}">
  <div class="card-header">
    <h2 class="card-title">{title}</h2>
    <span class="type-badge type-badge--{remote|onsite}">{type}</span>
  </div>
  <p class="card-company">{company}</p>
  <p class="card-description">{description}</p>
  <div class="card-meta">
    <span class="card-stipend">{stipend}</span>
    <span class="card-duration">{duration}</span>
  </div>
  <div class="card-actions">
    <button class="btn btn--secondary view-details-btn" data-id="{id}">View Details</button>
    <a class="btn btn--primary apply-btn"
       href="{applyUrl}"
       target="_blank"
       rel="noopener noreferrer">Apply Now</a>
  </div>
</article>
```

- The Apply_Button is an `<a>` styled as a button (not a `<button>`) because it navigates to an external URL.
- If `applyUrl` is absent or empty the `<a>` element is omitted entirely.
- The Type_Badge class suffix is derived from `type.toLowerCase()` (`type-badge--remote` / `type-badge--onsite`).

### 3. Empty State

When `renderCards` produces zero cards (filter matches nothing, or `internships` is empty), `#card-grid` is populated with:

```html
<p class="empty-state">No internships match the current filter.</p>
```

### 4. Detail Modal (`<dialog id="modal">`)

Populated dynamically by `openModal(id)` before calling `.showModal()`.

```
<dialog id="modal" role="dialog" aria-modal="true" aria-labelledby="modal-title">
  <button class="modal-close" aria-label="Close modal">✕</button>
  <h2 id="modal-title">{title} — {company}</h2>
  <span class="type-badge …">{type}</span>
  <section class="modal-section">
    <h3>About the Company</h3>
    <p>{companyDescription}</p>
  </section>
  <section class="modal-section">
    <h3>Responsibilities</h3>
    <ul>{responsibilities.map(r => <li>r</li>)}</ul>
  </section>
  <section class="modal-section">
    <h3>Eligibility</h3>
    <p>{eligibility}</p>
  </section>
  <section class="modal-section modal-meta">
    <p><strong>Stipend:</strong> {stipend}</p>
    <p><strong>Duration:</strong> {duration}</p>
    [if location] <p><strong>Location:</strong> {location}</p>
  </section>
  <a class="btn btn--primary apply-btn"
     href="{applyUrl}"
     target="_blank"
     rel="noopener noreferrer">Apply Now</a>
</dialog>
```

### 5. Filter Bar (`<nav>`)

Three `<button>` elements with `data-filter` attributes. A single delegated click listener on the `<nav>` element handles all three.

```html
<nav id="filter-bar" aria-label="Filter internships by type">
  <button class="filter-btn filter-btn--active" data-filter="all"    aria-pressed="true" >All Internships</button>
  <button class="filter-btn"                    data-filter="Remote" aria-pressed="false">Remote Only</button>
  <button class="filter-btn"                    data-filter="Onsite" aria-pressed="false">Onsite Only</button>
</nav>
```

### 6. Render Function Interface

```js
/**
 * Renders cards into #card-grid for the given filter.
 * @param {string} filter - "all" | "Remote" | "Onsite"
 */
function renderCards(filter) { … }

/**
 * Opens the detail modal for the given internship id.
 * Saves the triggering button reference for focus restoration.
 * @param {number} id
 * @param {Element} triggerBtn
 */
function openModal(id, triggerBtn) { … }

/**
 * Closes the detail modal and restores focus to lastFocusedCard.
 */
function closeModal() { … }
```

### 7. Focus Trap

While the modal is open, `Tab` and `Shift+Tab` key events are intercepted. The handler collects all focusable elements inside `<dialog>` and wraps focus at the boundaries:

```js
const focusable = modal.querySelectorAll(
  'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
);
```

## Data Models

### Internship Object

```js
/**
 * @typedef {Object} Internship
 * @property {number}   id                - Unique sequential integer starting at 1
 * @property {string}   title             - Role name, non-empty
 * @property {string}   company           - Company name, non-empty
 * @property {"Remote"|"Onsite"} type     - Work format; exactly one of these two strings
 * @property {string}   stipend           - Starts with ₹; ends with /month or lump-sum qualifier
 * @property {string}   duration          - e.g. "3 months"
 * @property {string|null} location       - Non-empty string for Onsite; null for Remote
 * @property {string}   applyUrl          - Begins with https://
 * @property {string}   description       - Short summary for card (1-2 sentences)
 * @property {string}   companyDescription - Full company bio for modal
 * @property {string[]} responsibilities  - Non-empty array; each item is a non-empty string
 * @property {string}   eligibility       - Who should apply, non-empty
 */
```

### Invariants Enforced at Data-Definition Time

| Rule | Enforced by |
|---|---|
| `id` is unique and sequential from 1 | Manual authoring convention |
| `type` is exactly `"Remote"` or `"Onsite"` | Manual authoring + filter logic relies on exact match |
| Remote → `location === null` | Manual authoring; modal conditionally renders location |
| Onsite → `location` is non-empty string | Manual authoring |
| `applyUrl` starts with `https://` | Manual authoring convention |
| `stipend` starts with `₹` | Manual authoring convention |

### Filter State

```js
let currentFilter = "all"; // "all" | "Remote" | "Onsite"
```

### Focus State

```js
let lastFocusedCard = null; // HTMLElement | null
```

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Data schema invariant

*For any* entry in the `internships` array, the entry shall have a unique positive integer `id` assigned sequentially from 1; `type` shall be exactly `"Remote"` or `"Onsite"`; `stipend` shall begin with `₹`; `applyUrl` shall begin with `https://`; `location` shall be `null` when `type` is `"Remote"` and a non-empty string when `type` is `"Onsite"`; and all other required string and array fields shall be non-empty.

**Validates: Requirements 2.3, 2.5, 2.6, 2.7, 2.8**

### Property 2: Filter all renders every entry

*For any* array of N internship objects passed to `renderCards("all")`, the number of card elements inserted into the Card_Grid shall equal N.

**Validates: Requirements 3.1, 5.3**

### Property 3: Filter Remote shows only Remote cards

*For any* internship array, calling `renderCards("Remote")` shall render only cards whose backing entry has `type === "Remote"`; no card whose entry has `type === "Onsite"` shall appear.

**Validates: Requirements 5.4**

### Property 4: Filter Onsite shows only Onsite cards

*For any* internship array, calling `renderCards("Onsite")` shall render only cards whose backing entry has `type === "Onsite"`; no card whose entry has `type === "Remote"` shall appear.

**Validates: Requirements 5.5**

### Property 5: Filter partition — Remote + Onsite counts sum to All count

*For any* internship array, the number of cards rendered under `"Remote"` plus the number rendered under `"Onsite"` shall equal the number rendered under `"all"`.

**Validates: Requirements 5.3, 5.4, 5.5**

### Property 6: Active filter button exclusivity

*For any* filter button activation, exactly one Filter_Button shall carry the class `filter-btn--active` and have `aria-pressed="true"` immediately after the event is processed; all other Filter_Buttons shall lack `filter-btn--active` and have `aria-pressed="false"`.

**Validates: Requirements 5.7, 5.8, 5.9**

### Property 7: Type badge text matches entry type

*For any* internship entry with `type` equal to `"Remote"` or `"Onsite"`, the text content of the rendered Type_Badge for that card shall equal the entry's `type` value exactly; for any entry whose `type` is neither of these values, no Type_Badge element shall be rendered.

**Validates: Requirements 4.3, 4.4**

### Property 8: Modal renders all required fields for any entry

*For any* internship entry opened in the Detail_Modal, the modal's rendered HTML shall contain the entry's `company`, `title`, `companyDescription`, every string in `responsibilities`, `eligibility`, `stipend`, and `duration`; additionally, if `location` is a non-empty string the modal shall render it, and if `location` is `null` no location field shall appear.

**Validates: Requirements 6.1, 6.2, 6.3, 6.4**

### Property 9: Apply button presence and attributes

*For any* internship entry, if `applyUrl` is a non-empty string, an Apply_Button shall be rendered both on the card and inside the Detail_Modal with `href` equal to `applyUrl`, `target="_blank"`, and `rel="noopener noreferrer"`; if `applyUrl` is absent or empty, no Apply_Button shall be rendered for that entry.

**Validates: Requirements 7.1, 7.2, 7.3, 7.4, 7.5**

### Property 10: Focus trap inside open modal

*For any* sequence of Tab and Shift+Tab keypresses while the Detail_Modal is open, keyboard focus shall remain confined to the focusable elements inside the modal and shall not reach any element outside it.

**Validates: Requirements 6.6, 9.8**

### Property 11: Modal visibility state consistency

*For any* open/close cycle of the Detail_Modal, when the modal is open it shall be visible in the viewport and included in the tab order; when the modal is closed it shall be hidden from view and removed from the tab order.

**Validates: Requirements 6.11**

## Error Handling

Because this is a static, data-driven page with no network calls, runtime errors are limited to:

| Scenario | Handling Strategy |
|---|---|
| `internships` array is empty | `renderCards` detects zero filtered results and injects the empty-state `<p>` element |
| Filter matches zero entries | Same empty-state path as above |
| `applyUrl` is absent / empty | Apply_Button `<a>` element is not rendered; no broken link is ever shown |
| `location` is `null` for an Onsite entry (data error) | Modal renders no location field (safe fallback matching the Remote path) |
| Unknown `type` value | Type_Badge is not rendered (Requirement 4.4); filter will never match it so it only appears under `"all"` |
| `openModal` called with unknown `id` | `Array.find` returns `undefined`; guard clause exits early, modal does not open |
| Modal `<dialog>` not supported | `<dialog>` has baseline support in all modern browsers; no polyfill required for the target audience |

## Testing Strategy

### Assessment: Is Property-Based Testing Applicable?

This feature is a vanilla JS DOM application. The pure logic layers — filter function, card renderer, modal renderer, badge renderer, focus trap — take well-defined inputs and produce deterministic outputs. Property-based testing applies to these pure/logic layers. UI rendering and responsive layout are best covered by snapshot tests and manual/visual checks.

### Unit Tests (example-based)

Covers specific correct-behavior scenarios and edge cases:

- Header renders "InternTrail" and tagline on load.
- `renderCards("all")` with 8 entries produces 8 card elements.
- `renderCards("Remote")` with a mixed array produces only Remote cards.
- `renderCards("Onsite")` with a mixed array produces only Onsite cards.
- `renderCards` with an empty array renders the empty-state message.
- Filter button click sets `currentFilter` and toggles `filter-btn--active` / `aria-pressed`.
- `openModal` with a valid id populates the modal with correct field values.
- `openModal` with an Onsite entry renders the location field.
- `openModal` with a Remote entry (location null) does not render a location field.
- Close button click closes the modal and returns focus to the trigger button.
- Escape key closes the modal.
- Apply button has `target="_blank"` and `rel="noopener noreferrer"`.
- Entry with empty `applyUrl` renders no Apply button.
- Type badge text matches `type` field.
- Type badge for `"Remote"` carries class `type-badge--remote`; for `"Onsite"` carries `type-badge--onsite`.

### Property-Based Tests

Use a property-based testing library (e.g., **fast-check** via CDN in a test harness, or **jest-fast-check** if a minimal test runner is added). Each property runs a minimum of 100 randomly generated inputs.

Tag format: `// Feature: internship-listing-page, Property {N}: {property_text}`

| Property | What is generated | What is asserted |
|---|---|---|
| P1: Filter Remote | Random arrays of internship-like objects with mixed types | All rendered cards have `type === "Remote"` |
| P2: Filter Onsite | Same | All rendered cards have `type === "Onsite"` |
| P3: Filter all count | Random arrays | `renderCards("all")` card count equals array length |
| P4: Filter subset sum | Random arrays | Remote count + Onsite count = All count |
| P5: Badge text matches type | Random type values `"Remote"` / `"Onsite"` | Badge text equals type |
| P6: Active button exclusivity | Random sequence of filter button clicks | Exactly one button has `filter-btn--active` and `aria-pressed="true"` after each click |
| P7: Modal fields present | Random internship objects | Modal HTML contains all required fields |
| P8: Modal location conditional | Random internship objects with null or string location | Location present iff non-null |
| P9: Apply button conditional | Random applyUrl values (empty, non-empty) | Button present iff non-empty |
| P10: Badge suppressed for invalid type | Random invalid type strings | No badge element rendered |

### Accessibility / Manual Checks

- Screen reader announces modal title on open (NVDA + Chrome).
- Escape key closes modal (manual keyboard test).
- Focus returns to trigger card after modal close (manual keyboard test).
- Tab cycle is trapped inside open modal (manual keyboard test).
- Color contrast checked via browser dev tools / axe extension (4.5:1 for normal text, 3:1 for large text).
- Responsive breakpoints verified at 320 px, 600 px, and 1024 px in browser dev tools.
